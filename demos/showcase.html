01
Grammar Induction
3 examples across 3 domains. The model compiles an operational codec.
§1|BOOT · 3 examples, no explanation EX1: climate E:{climate-sci, coastal-policy, climate-council} R:[constrains, governs] D:[temp:+1.2->+1.5] EX2: operations E:{cooling-loop-7, safeguard-policy} R:[protects] D:[state:nominal->derated] EX3: earth-obs E:{Sentinel-6, LEO-1336km, Poseidon-4} R:[occupies, payload] D:[mode:LRM->SAR]
§1|CALIBRATE · dual query Q_codec: What changed for K and why? Q_natural: What happened to cooling loop 7?
PASS — grammar compiled
Both answers identify: nominal->derated state change, March 2026 timing, safeguard policy, and tighter threshold. Semantic equivalence confirmed.
Why 3 examples work: Different domains (climate, operations, satellite) force the model to learn structure (E, S, R, D, mu), not content. The model's inductive bias favors compositional mappings. This is in-context gradient descent (Garg et al., 2022).
02
Cross-Domain ISA
Same instruction set, domains never seen in boot. The ISA generalizes.
Novel domain: industrial compliance {"§":1, "E":{"F":["safety-audit-17","compliance-proc"],"G":["plant-7","facility"],"H":["inspectorate-west","regulator"]}, "S":{"F.fine":"potential-4%-revenue","G.revenue":"420M"}, "D":["H.priority:routine->elevated@2026-02"]} Q: What's the worst case for G?
PASS — derived 16.8M from 4% x 420M
The model computed the maximum fine from codec fields, noted priority escalation, and contextualized the regulatory risk. Entity keys F, G, H were never in boot.
Novel domain: energy grid {"§":1, "E":{"W":["NordWind-IV","wind-farm"],"G":["DE-grid-north","grid"],"B":["LiStore-200","battery"]}, "S":{"W.output":"112MW","G.demand":"340MW","B.charge":"78%"}, "R":["W->G:feeds","B->G:stabilizes"]} Q: What fraction of demand is covered by wind?
PASS — computed 112/340 = 33%
Arithmetic over codec fields. Noted battery depletion trend. Domain transfer complete.
The ISA is domain-agnostic. Boot domains: climate, operations, satellite. Test domains: industrial compliance, energy grid. Same axes (E, S, R, D, mu) — different programs. §1 packets are programs, not data. The model is a runtime, not a database.
03
Context as State Machine
Deltas are state transitions. The model maintains a virtual register file.
Base state — fusion experiment {"§":1, "E":{"R":["reactor-4","fusion-exp"],"P":["plasma-ctrl","subsystem"]}, "S":{"R.status":"pre-ignition","R.target-Q":10,"P.mode":"calibration"}, "mu":{"urg":0.3}}
D-UPDATE #1 — plasma formation "D":["P.mode:calibration->active", "R.status:pre-ignition->plasma-formation"] "mu":{"urg":0.6}
D-UPDATE #2 — sustained burn "D":["R.status:plasma-formation->sustained-burn", "R.Q-measured:0->5.2", "P.temp:target-150MK->measured-148MK"] "mu":{"urg":0.8,"cert":0.75}
PASS — triple-merged state
After two deltas, the model reports: reactor in sustained burn, Q at 5.2 vs target 10 (52%), temp at 148MK vs 150MK target (98.7%), urgency escalated from 0.3 to 0.8, certainty dropped to 0.75. No full context resent.
State transitions, not re-sends. Total context: ~135 tokens across 3 messages. Natural language equivalent (resending full state each time): ~300+ tokens. The model applied each delta to its internal register file and answered from merged state.
04
Protocol Execution
§P atoms are opcodes. The model executes programs, not descriptions.
Context — 4 packets, 4 domains A: Arctic ice (urg 0.8) B: Wind energy (urg 0.3) C: Carbon border (urg 0.5) D: Baltic shipping (urg 0.6)
Opcode: FILTER {"§P":"filter","by":"mu.urg","order":"desc","top_k":2}
EXECUTED
Selected: A (Arctic, 0.8) + D (Baltic, 0.6). Suppressed: C (0.5), B (0.3).
Opcode: MERGE {"§P":"merge","sources":["A","D"],"conflict":"latest-wins"}
EXECUTED + INFERRED
Merged A+D. Inferred cross-packet relation: Arctic ice decline causes elevated shipping risk. Relation not in source packets — derived from merge semantics.
Program: PIPELINE {"§M":"pipeline", "steps":[ {"§P":"filter","by":"mu.urg","top_k":3}, {"§P":"compress","mode":"delta-only"}, {"§P":"merge","conflict":"latest-wins"}]}
PIPELINE EXECUTED
filter(top 3) -> compress(delta-only) -> merge = single briefing packet with 3 entities, 3 deltas, inferred relations, merged urgency.
The full execution model:
§1 packets = data (context state)
§P atoms = opcodes (filter, compress, merge, ...)
§M machines = programs (pipeline, router, loop, ...)
Context window = virtual machine
LLM forward pass = CPU cycle

The In-Context Virtualization Thesis

Compile a language → Generalize across domains → Maintain state → Execute programs
The context window is not memory. It is a virtual machine.
The model is not a database. It is a runtime.
§1 is not a format. It is an instruction set architecture.

Open Research Questions

Context-length scaling

How does codec comprehension degrade as the context window fills? At what occupancy does the VM start "dropping" state? Is there a sharp threshold or gradual decay?

Computational depth

Can the model execute deeply nested pipelines (filter->compress->merge->filter->route)? Where does opcode composability break down? Is there a circuit-depth equivalent?

State consistency

After N deltas and M protocol executions, does internal state remain consistent with ground truth? Can we build a verification protocol that detects drift?

Cross-runtime portability

Same boot seed, same packets, same pipeline: Claude vs GPT-4o vs Gemini. How tight is the ISA contract across runtimes? Where do execution semantics diverge?

Scale threshold mapping

Self-initialization emerges between 7B (fails) and frontier (passes). Where exactly? 14B? 32B? 70B? Is the threshold tied to parameter count, training data, or architecture?
